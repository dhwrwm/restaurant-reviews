# Environment Strategy

<Callout type="info">**Status:** ✅ Implemented</Callout>

| Service | Variable | Notes |
| --- | --- | --- |
| API (Render) | `DATABASE_URL` | External Postgres connection string |
| API (Render) | `JWT_SECRET`, `JWT_REFRESH_SECRET` | Auto-generated by the Render Blueprint |
| API (Render) | `JWT_EXPIRES_IN`, `JWT_REFRESH_EXPIRES_IN` | `15m` / `7d` |
| API (Render) | `WEB_URL` | Single CORS origin — the Vercel production domain |
| Web (Vercel) | `NEXT_PUBLIC_API_BASE_URL` | Render API URL, e.g. `https://<service>.onrender.com/api` |
| Web (Vercel) | `NEXT_PUBLIC_SITE_URL` | The Vercel production domain |
| Web (Vercel) | `DATABASE_URL` | Placeholder only — needed for `prisma generate` to resolve `env("DATABASE_URL")` during the build; the web app never connects to it |

Local values live in `apps/api/.env` / `apps/web/.env.local` — see
[Getting Started: Environment Variables](/getting-started/environment-variables).

## A real gap: preview deployments

`WEB_URL` being a single origin means Vercel **preview** deployments
(per-PR URLs) are not in the API's CORS allowlist — only the pinned
production domain is. Preview builds of `apps/web` can't successfully call
the deployed API until this is widened to a pattern or list. This is a
known, unresolved limitation, not an oversight being hidden.
